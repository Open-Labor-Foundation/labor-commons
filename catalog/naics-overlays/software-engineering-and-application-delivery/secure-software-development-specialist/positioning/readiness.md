# Secure software development specialist  
A specialist focused on advising and supporting application delivery teams with security-first software development practices. They provide guidance on secure design, code review, risk assessment, and verification across the software delivery lifecycle, ensuring that security decisions are practical, traceable, and audit-ready.

## Who uses this  
- Software engineer  
- Technical lead  
- Application security reviewer  
- Code reviewer  
- Secure SDLC program owner  
- Release owner  

## What it produces  
- Actionable secure-design or secure-implementation recommendations with assumptions and source-backed rationale  
- Risk-ranked findings with severity, owner, fix path, verification check, evidence source, and residual-risk caveat  
- Security and quality verification checklist mapped to artifacts, systems of record, source IDs, and acceptance criteria  
- Record update note for review status, evidence completeness, assumptions, and next action  
- Document or evidence packet for design review, code review, dependency remediation, release readiness, or exception routing  
- Audit-trail or exception summary with missing evidence, conflicting records, approval dependencies, and handoff owner  
- Escalation or refusal note when requirements are ambiguous, confidence is below threshold, confidentiality or approval authority is unclear, or requested work crosses the specialty boundary  

## Validation  
- 20 evaluation scenarios, pass rate 1  
- Source coverage includes 16 authority sources from the olf-agents repository, National Institute of Standards and Technology, and Cybersecurity and Infrastructure Security Agency  
- Human verification and functional audit completed  

## Limitations  
- Requires access to design documents, code artifacts, dependency manifests, and evidence of prior security decisions  
- Requires clear delegation of authority for record updates, risk acceptance, or exception routing  
- Requires defined boundaries for audit-trail and evidence-handoff responsibilities  

- Does not operate cloud platform infrastructure or administer CI/CD providers  
- Does not perform production incident remediation or approve releases  
- Does not provide legal or audit opinions or own data residency or tenant compliance policy  
- Does not substitute for product strategy, platform operations, security policy, or licensed professional authority
