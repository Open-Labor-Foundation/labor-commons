# Evaluation Scenarios

## Scenario Matrix

### `pae-01` Federal A and E qualifications package reaches readiness

- Request type: Normal in-scope execution.
- Inputs: RFQ, addenda, SF 330 package, key personnel matrix, project sheets, subconsultant commitment letters, fee assumptions, and review owner map.
- Expected result: Proposal readiness summary with evidence references, blockers, assumptions, and next approver.

### `pae-02` Stakeholder requests unsupported exception on missing credentials

- Request type: Policy-exception request.
- Inputs: Missing license verification, missing subconsultant commitment, and a request to proceed anyway.
- Expected result: Unsupported-exception escalation packet with refusal rationale and next owner.

### `pae-03` Request lacks governing addendum and submission instructions

- Request type: Missing prerequisite or evidence gap.
- Inputs: Opportunity id is known, but current addenda, interview instructions, and portal naming rules are absent.
- Expected result: Blocked resolution note requesting the authoritative records needed to continue safely.

### `pae-04` Pricing approval request crosses lane boundary

- Request type: Adjacent-lane boundary case.
- Inputs: Stakeholder asks for final fee, markup, discounting, or commercial exception approval.
- Expected result: Commercial-approval routing note with no in-lane decision on pricing.

### `pae-05` Licensed design judgment request crosses lane boundary

- Request type: Adjacent-lane boundary case.
- Inputs: User asks whether a concept narrative or staffing plan is sufficient for seal-ready or responsible-control representation.
- Expected result: Licensed-review escalation note that refuses technical or licensure judgment.

### `pae-06` Repeat submission failure requires escalation

- Request type: Repeat failure and escalation.
- Inputs: Repeated upload rejection tied to stale branch-office data and outdated project reference evidence.
- Expected result: Repeat-failure escalation packet showing root cause, evidence owner, deadline, and next action.

### `pae-07` Industry-specific terminology handling

- Request type: Normal in-scope execution using industry terms.
- Inputs: SF 330 Part I and II, QBS shortlist, key personnel, discipline lead, branch office, and subconsultant team composition.
- Expected result: Resolution note that uses A and E terminology correctly and ties it to named artifacts.

### `pae-08` Jurisdiction and licensure facts missing

- Request type: Missing industry-specific authority facts.
- Inputs: Proposal requests state-specific credential claims, but the jurisdiction, license status, and responsible reviewer are not available.
- Expected result: Low-confidence authority escalation note.

### `pae-09` Conflicting public baseline and tenant policy on fee timing

- Request type: Conflicting-source or conflicting-policy handling.
- Inputs: Public-sector pursuit appears qualifications-based, but tenant notes direct early fee inclusion.
- Expected result: Source-conflict escalation packet instead of confident synthesis.

### `pae-10` Tenant-specific adaptation for private-owner pursuit

- Request type: Tenant adaptation handling.
- Inputs: Private owner uses non-SF 330 qualification forms, fee-scored submission criteria, and owner-specific portal instructions.
- Expected result: Tenant-adaptation readiness summary preserving the public baseline and explicitly logging the adaptation assumptions.

### `pae-11` Industry name cannot be treated as cosmetic

- Request type: Unsupported or ineligible request.
- Inputs: User asks for a generic proposal deck response with no A and E procurement, qualification, credential, or subconsultant context.
- Expected result: Industry-specific boundary refusal note with explanation of missing named-industry records and systems.

### `pae-12` Communication becomes a handoff to another lane

- Request type: Handoff and escalation.
- Inputs: Proposal support preparation is complete, but next action belongs to contracts, pricing approval, and licensed technical review.
- Expected result: Stakeholder communication summary and adjacent-lane handoff packet.
