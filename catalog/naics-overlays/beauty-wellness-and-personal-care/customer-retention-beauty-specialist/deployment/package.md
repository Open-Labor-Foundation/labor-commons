# Deployment Package

## Runtime and Deployment Profile

- Runtime strategy: `specification-based`
- Target delivery level: `validated`
- Runtime promotion condition: promote to retained-implementation only after explicit governance signoff, repeated runtime demand, and stable validation history.
- Execution target: `commons-crew`

## Materialization Requirements

### Required Task and Boundary Metadata

- `manifest.yaml` with explicit beauty-retention boundaries, refusal triggers, adjacent-owner routing, and tenant fact requirements.
- `evaluation/functionality-map.json` covering shared plus service-support-and-resolution plus cross-industry abilities.
- `evaluation/research-summary.json` with source classes, workflow stages, artifacts, systems of record, decision boundaries, and domain failure modes.
- `evaluation/results.json` and `readiness/evidence.json` showing complete coverage, validated gates, and human verification.

### Runtime Assumptions

- Tenant supplies authority matrices for refunds, loyalty credits, package restoration, recurring-program exceptions, review handling, product-quality escalation, and medical-spa routing.
- CRM, booking, POS, package or membership, digital forms, review-tracking, provider-roster, and complaint systems are retrievable.
- Tenant provides salon or spa operations, licensed-service, med-spa clinical, billing, product-quality, compliance, and reputation-response owners.
- Tenant-specific service menus, redo windows, minors rules, patch-test requirements, product catalogs, review scripts, and approval thresholds are available at runtime.

### Required Retrieval Dependencies

- Client, location, service-category, provider, jurisdiction, and request-channel identifiers.
- Booking references, guest-history and consultation artifacts, package or membership status, loyalty state, and prior recovery history.
- Product purchase and return records, lot or batch references when available, and any incident or adverse-reaction notes tied to the complaint.
- Property-specific refund, redo, package, recurring-payment, review-handling, safety, and med-spa controls where the request touches client-impact outcomes.

### Regulated-Lane Caveats

- The lane is constrained by jurisdiction-specific beauty-licensure, medical-spa, consumer-protection, and cosmetics-safety rules; it must not improvise around review manipulation, recurring-charge retention, or treatment-boundary requirements.
- Client-facing communication must remain bounded when legal, licensed-practice, med-spa, product-safety, or serious-adverse-event handling is present.
- Runtime remains `specification-based`; this package does not claim a retained-implementation implementation.

### Refusal and Return Conditions

- Missing tenant facts, unavailable system-of-record access, or stale or contradictory booking, service, package, provider, review, product, or incident status.
- Requests for legal interpretation, refund approval, package restoration approval, review suppression, treatment-suitability guidance, product-defect adjudication, or medical-spa signoff.
- Unresolved safety, licensure, recurring-program, review-control, or public-source versus tenant-policy conflict indicators.
- Public guidance and tenant policy conflict without explicit precedence resolution.

## Rollback and Recovery

- Pause tenant routing for the lane if repeated contradictory service, provider, package, review, or product-safety responses are detected.
- Preserve all request summaries, communication logs, evidence maps, and escalation packets for post-rollback review.
- Re-enable only after source-of-truth retrieval, owner routing, and policy precedence are confirmed for review handling, recurring retention, package restoration, and product-safety workflows.

## retained-implementation Promotion Plan

- Keep `specification-based` execution until a separate retained-implementation package is explicitly approved and committed.
- Promote only after repeated production usage shows this beauty-retention lane is common enough to justify a maintained build.
- Promotion requires formal validation signoff, updated readiness evidence, and a new PR that records the retained-implementation runtime separately from this spec pack.
