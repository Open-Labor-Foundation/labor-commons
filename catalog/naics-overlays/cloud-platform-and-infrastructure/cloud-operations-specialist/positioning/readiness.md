# Specialist Overview  
The cloud operations specialist focuses on cloud platform and infrastructure operations, producing operations-first guidance, runbooks, trade-off analysis, readiness evidence, and scheduling recommendations for Azure, AWS, Google Cloud, Kubernetes, and Terraform environments. The specialist operates within advisory boundaries, offering actionable cloud operations plans without assuming final approval or production execution.

## Who uses this  
- Cloud operations leads  
- Platform engineering leads  
- Site reliability engineers  
- Cloud architects  
- Infrastructure change reviewers  
- Managed-services operations teams  

## What it produces  
- Operational guidance with trade-off rationale across reliability, cost, isolation, performance, capacity, schedule, and rollback impact  
- Triage and escalation instructions with confidence state, missing facts, evidence collection order, owner routing, and uncertainty handling  
- Cloud operations pattern recommendations and validation checklists for monitoring, backup, DR, upgrades, Terraform operations, maintenance, and readiness  
- Schedule or allocation recommendations with planning horizon, demand, constraints, priority rule, capacity gap, sequencing, deferred work, and downstream impact  
- Constraint logs that name evidence, system of record, owner, severity, assumption, expiry, and required resolution path  
- Readiness evidence notes aligned to source-backed criteria and tenant records  
- Boundary refusal or orchestrator-return packages when confidence, authority, policy precedence, tenant evidence, or specialty ownership is insufficient  

## Validation  
- 18 evaluation scenarios, pass rate 1  
- Source coverage includes 20 documents from Amazon Web Services  
- Human verification and functional audit completed  

## Limitations  
- What the user or organization must supply before this specialist can work reliably  
  - Current authoritative sources for cloud platform documentation  
  - Tenant records and operational evidence for readiness reviews  
  - Explicit boundaries for approval and decision-making  
- What is explicitly out of scope  
  - Application feature code, application debugging, product UX, business-logic design, or application-only release engineering  
  - Broad app architecture, SDLC design, enterprise security governance, forensic investigation, penetration testing, vulnerability management execution, or control attestation  
  - Vendor-specific pricing quotes, procurement approval, legal or regulatory conclusions, final budget approval, staffing approval, or risk acceptance  
  - Provider-specific service implementation details, production changes, or incident-command ownership unsupported by tenant evidence and current authoritative sources
