# Specialist Overview  
A cloud migration specialist provides infrastructure-focused guidance for moving workloads to public, hybrid, or Kubernetes-based cloud environments. It generates structured migration plans, validation evidence, readiness assessments, and operational runbooks while preserving clear boundaries between infrastructure decisions and application, compliance, or security ownership.

## Who uses this  
- Cloud migration architects  
- Platform engineering leads  
- Migration program managers  
- Infrastructure reviewers  
- Site reliability engineers  
- Managed-services migration teams  

## What it produces  
- Migration strategy memo with wave and sequencing recommendations  
- Cutover runbook with rollback checkpoints  
- Readiness evidence and test outcome summary  
- Deployment and commercial packaging notes  
- Schedule or allocation recommendation with planning horizon, demand, constraints, priority rule, capacity gap, sequencing, deferred work, and downstream impact  
- Constraint log with evidence, system of record, owner, severity, assumption, expiry, and required resolution path  
- Approval-required escalation note for tradeoffs outside lane authority  
- Boundary refusal or orchestrator-return package when confidence, authority, policy precedence, tenant evidence, or specialty ownership is insufficient  

## Validation  
- 20 evaluation scenarios, pass rate 1  
- Covers Amazon Web Services sources  
- Human verification and functional audit completed  

## Limitations  
- Requires a defined workload inventory, dependency map, and migration inventory with authoritative source freshness  
- Requires explicit migration planning horizon, approval boundaries, and tenant isolation constraints  
- Requires access to landing-zone architecture, network dependency readiness, and operational handoff criteria  
- Does not own application feature development, UI engineering, business-logic redesign, database schema redesign, or code architecture refactors  
- Does not own legal/compliance rulings, cybersecurity incident response, vulnerability management, penetration testing, or formal control attestation  
- Does not own identity product administration, endpoint administration, or provider-specific production implementation outside migration guidance  
- Does not present migration schedules, staffing, budget, or risk recommendations as final approval
